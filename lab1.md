## 1. Раздел Management (Управление)

### 1.1. Server Status (Состояние сервера)

В разделе отображается общая информация о сервере и подключении к нему. Информация логически сгруппирована. Можно выделить следующие группы:

- **a.** Общая информация (например, название хоста, номер порта, версия БД).
- **b.** Настройки сервера (например, включен ли брандмауэр, используется ли SSL).
- **c.** Каталоги сервера.
- **d.** Сводка по используемым ресурсам компьютера (ОЗУ, процессор и т. д.).
- **e.** Настройки соединения SSL (если SSL включена).

---

### 1.2. Data Export (Экспорт данных)

Раздел предназначен для создания резервных копий баз данных и их объектов. Интерфейс логически сгруппирован:

- **a.** **Выбор объектов для экспорта** — список доступных схем (БД) с возможностью выбора конкретных таблиц, отображений, хранимых процедур и функций.
- **b.** **Формат экспорта** — выбор формата выгрузки:
  - Дамп-файл (единый SQL-файл с инструкциями создания и заполнения);
  - Отдельные SQL-файлы для каждого объекта;
  - Файлы с данными в формате CSV (только данные таблиц).
- **c.** **Параметры экспорта** — дополнительные настройки (включать ли CREATE-инструкции, DROP-инструкции, кодировку, путь для сохранения, сжатие файла и т. д.).
- **d.** **Журнал выполнения** — область с информацией о процессе экспорта (прогресс, ошибки, предупреждения).

---

### 1.3. Data Import and Restore (Импорт и восстановление данных)

Раздел предназначен для загрузки ранее созданных резервных копий обратно в сервер БД. Информация сгруппирована:

- **a.** **Выбор источника данных** — указание пути к файлу дампа (SQL или CSV).
- **b.** **Параметры импорта** — настройка поведения при восстановлении:
  - Создание новой схемы или использование существующей;
  - Режим обработки ошибок (продолжать выполнение или прерывать);
  - Режим очистки существующих данных перед импортом.
- **c.** **Формат импорта** — выбор типа загружаемого файла (SQL-дамп или CSV-файл с данными).
- **d.** **Журнал выполнения** — область с информацией о процессе импорта (прогресс, ошибки, предупреждения).

---

### 1.4. Users and Privileges (Пользователи и привилегии)

Раздел предназначен для управления учётными записями MySQL и их правами доступа. Информация сгруппирована:

- **a.** **Список пользователей** — отображение всех учётных записей, зарегистрированных на сервере (имя пользователя, хост, тип аутентификации).
- **b.** **Управление учётными записями** — добавление новых пользователей, удаление существующих, изменение имени или хоста.
- **c.** **Настройки аутентификации** — задание или смена пароля, выбор метода аутентификации (Native MySQL, SHA256, LDAP и др.), настройка срока действия пароля.
- **d.** **Administrative Roles (Административные роли)** — назначение глобальных привилегий (SELECT, INSERT, UPDATE, DELETE, CREATE, DROP, SUPER, SHUTDOWN и т. д.).
- **e.** **Schema Privileges (Привилегии схем)** — детализация прав доступа к конкретным базам данных и их объектам.

---

### 1.5. Management (Администрирование / Служебные операции)

В некоторых версиях Workbench присутствует дополнительный подраздел со служебными операциями, сгруппированными по назначению:

- **a.** **Работа с логами** — просмотр журналов ошибок (Error Log), медленных запросов (Slow Query Log), бинарных логов.
- **b.** **Управление процессами** — просмотр списка активных соединений и выполняющихся запросов (Process List), возможность принудительного завершения «зависших» процессов (Kill).
- **c.** **Работа с переменными сервера** — просмотр и изменение системных переменных MySQL в режиме реального времени.
- **d.** **Статистика** — просмотр накопленной статистики по запросам, командам, использованию индексов и т. д.

---

> **Примечание:** Конкретный набор и название пунктов раздела Management может незначительно отличаться в зависимости от версии MySQL Workbench.

## 2. Раздел Instance (Экземпляр БД)

### 2.1. Startup / Shutdown (Запуск / Остановка)

В разделе отображается информация о текущем состоянии MySQL-сервера и предоставляются элементы управления его жизненным циклом. Информация логически сгруппирована:

- **a.** Текущее состояние сервера (запущен / остановлен).
- **b.** Кнопки управления: запуск сервера (**Start Server**), остановка сервера (**Stop Server**).
- **c.** Лог-файл сервера (Server Log) — вывод сообщений, появляющихся при запуске и остановке, что позволяет диагностировать ошибки.
- **d.** Информация о конфигурации запуска (команда запуска, путь к конфигурационному файлу).

---

### 2.2. Options File (Файл параметров конфигурации)

В разделе осуществляется просмотр и редактирование конфигурационного файла MySQL (`my.cnf` / `my.ini`) через графический интерфейс без необходимости ручного редактирования текстового файла. Настройки сгруппированы по категориям:

- **a.** **General** (Общие) — общие параметры сервера (базовый каталог данных, порт по умолчанию, ID сервера).
- **b.** **Networking** (Сеть) — сетевые параметры (адрес привязки, порт, максимальное количество соединений, тайм-ауты).
- **c.** **Security** (Безопасность) — параметры безопасности (SSL-сертификаты, режим безопасного обновления и др.).
- **d.** **Advanced** (Расширенные) — тонкие настройки (размеры буферов, кэшей, параметры движка InnoDB, бинарного лога и др.).
- **e.** Для каждой опции указывается её текущее значение, и возможна фильтрация / поиск по названию параметра.

---

## 3. Раздел Performance (Производительность)

### 3.1. Performance Dashboard (Панель производительности)

Раздел предоставляет сводную визуальную панель мониторинга производительности MySQL-сервера в режиме реального времени. Информация представлена в виде графиков и индикаторов, сгруппированных по блокам:

- **a.** **Network Traffic** (Сетевой трафик) — объём данных, отправляемых и получаемых сервером.
- **b.** **MySQL Connections** (Соединения MySQL) — количество активных и общее количество соединений.
- **c.** **InnoDB I/O Activity** (I/O-активность InnoDB) — операции чтения/записи движка InnoDB.
- **d.** **InnoDB Buffer Usage** (Использование буфера InnoDB) — степень использования пула буферов InnoDB.
- **e.** **Query Activity** (Активность запросов) — количество выполняемых запросов в секунду.

---

### 3.2. Performance Reports (Отчёты по производительности)

В разделе доступны детализированные отчёты, позволяющие анализировать отдельные аспекты работы сервера. Отчёты сгруппированы по категориям:

- **a.** **General** — общая информация: состояние сервера, список активных процессов (Process List).
- **b.** **InnoDB** — отчёты, связанные с движком InnoDB: использование буферного пула, транзакции, блокировки, I/O-операции.
- **c.** **Memory** — отчёты по использованию оперативной памяти различными компонентами сервера.
- **d.** **Performance Schema** — отчёты на основе данных, собранных Performance Schema (наиболее «тяжёлые» запросы, время ожидания, использование индексов и т. д.).

---

### 3.3. Performance Schema Setup (Настройка Performance Schema)

Раздел предназначен для тонкой настройки инструментария **Performance Schema**, который собирает метрики о работе сервера. Управление сгруппировано:

- **a.** **Instruments** (Инструменты) — список типов событий, для которых может вестись сбор данных (ожидания, стадии выполнения, транзакции, ошибки и др.). Для каждого инструмента можно включить или отключить сбор.
- **b.** **Consumers** (Потребители) — определение того, в какие таблицы Performance Schema будут записываться собранные данные (текущие, история, длинная история).
- **c.** Изменения применяются к работающему серверу и могут быть сохранены в конфигурационный файл для сохранения между перезапусками.

---

> **Примечание:** Конкретный набор доступных пунктов меню может незначительно отличаться в зависимости от версии MySQL Workbench и операционной системы, на которой он установлен.

## Задание 5. Модифицированная таблица `users`

**Итоговая структура таблицы:**

| № | Поле | Тип | Атрибуты | Значение по умолчанию | Комментарий |
|---|---|---|---|---|---|
| 1 | `id` | INT | PK, NOT NULL, AI | — | Автоинкрементный идентификатор |
| 2 | `name` | VARCHAR(50) | NULL | NULL | Имя пользователя |
| 3 | `email` | VARCHAR(45) | NULL | NULL | Email |
| 4 | `gender` | ENUM('M','F') | NULL | NULL | Пол: M — мужской, F — женский |
| 5 | `bday` | DATE | NULL | NULL | Дата рождения |
| 6 | `postal_code` | VARCHAR(10) | NULL | NULL | Почтовый индекс |
| 7 | `rating` | FLOAT | NULL | NULL | Рейтинг пользователя |
| 8 | `created` | TIMESTAMP | NOT NULL | CURRENT_TIMESTAMP | Дата создания записи |

### Описание поля `created`

Тип данных `TIMESTAMP` с выражением по умолчанию `CURRENT_TIMESTAMP()` означает, что при вставке новой записи в это поле **автоматически записывается текущая дата и время сервера**. Это позволяет фиксировать момент создания каждой записи без участия пользователя. Если не указано `ON UPDATE CURRENT_TIMESTAMP`, при последующих обновлениях строки значение поля не изменяется.

### Поля, допускающие NULL

С точки зрения пользователя, который не хочет делиться персонализированной информацией, значение `NULL` (отсутствие данных) могут иметь следующие поля:

- `name` — пользователь может не указывать имя;
- `email` — пользователь может не оставлять почту;
- `gender` — пол не является обязательным;
- `bday` — дату рождения можно скрыть;
- `postal_code` — почтовый индекс можно не сообщать;
- `rating` — рейтинг может быть ещё не присвоен.

Поле `created` **не может** содержать `NULL`, так как момент создания записи — это служебная информация, генерируемая СУБД автоматически.

**SQL-запрос модификации таблицы:**

```sql
ALTER TABLE `simpledb`.`users` 
  ADD COLUMN `gender` ENUM('M', 'F') NULL AFTER `email`,
  ADD COLUMN `bday` DATE NULL AFTER `gender`,
  ADD COLUMN `postal_code` VARCHAR(10) NULL AFTER `bday`,
  ADD COLUMN `rating` FLOAT NULL AFTER `postal_code`,
  ADD COLUMN `created` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP AFTER `rating`;
```

## Задание 8. Создание таблицы `resume` с внешним ключом

**Структура таблицы:**

| Поле | Тип | Атрибуты |
|---|---|---|
| `resumeid` | INT | PK, NOT NULL, AUTO_INCREMENT |
| `userid` | INT | NOT NULL, FOREIGN KEY → `users.id` |
| `title` | VARCHAR(100) | NOT NULL |
| `skills` | TEXT | NULL |
| `created` | TIMESTAMP | NOT NULL, DEFAULT CURRENT_TIMESTAMP |

**SQL-запрос создания таблицы:**

```sql
CREATE TABLE `simpledb`.`resume` (
  `resumeid` INT NOT NULL AUTO_INCREMENT,
  `userid` INT NOT NULL,
  `title` VARCHAR(100) NOT NULL,
  `skills` TEXT NULL,
  `created` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`resumeid`),
  INDEX `fk_resume_users_idx` (`userid` ASC) VISIBLE,
  CONSTRAINT `fk_resume_users`
    FOREIGN KEY (`userid`)
    REFERENCES `simpledb`.`users` (`id`)
    ON DELETE CASCADE
    ON UPDATE CASCADE
) ENGINE = InnoDB DEFAULT CHARACTER SET = utf8 COLLATE = utf8_general_ci;
```

### Поведение СУБД при каскадных операциях

- **`ON DELETE CASCADE`** — при удалении пользователя из таблицы `users` **все связанные с ним резюме** в таблице `resume` будут **автоматически удалены**.
- **`ON UPDATE CASCADE`** — при изменении значения `id` пользователя в таблице `users` значения `userid` во всех связанных записях таблицы `resume` будут **автоматически обновлены** на новое значение.

**Вывод:** добавление записи с несуществующим `userid` **невозможно**. Ограничение внешнего ключа (`FOREIGN KEY`) обеспечивает **целостность данных** и не позволяет создать резюме, привязанное к несуществующему пользователю.

## Задание 10. Удаление пользователей и каскадные операции

### Удаление пользователей, имеющих связанные резюме

**SQL-запросы на удаление:**

```sql
DELETE FROM `simpledb`.`users` WHERE (`id` = '1');
DELETE FROM `simpledb`.`users` WHERE (`id` = '2');
```

Поскольку при создании внешнего ключа было указано **`ON DELETE CASCADE`**, СУБД **автоматически удалила** все резюме, связанные с удалёнными пользователями:

- Удалён пользователь `id = 1` → автоматически удалены его резюме `resumeid = 1` и `resumeid = 3`.
- Удалён пользователь `id = 2` → автоматически удалено его резюме `resumeid = 2`.

В таблице `resume` не осталось «сиротских» записей — целостность данных сохранена.

Выполним запрос на обновление первичного ключа:

```sql
UPDATE `simpledb`.`users` SET `id` = 100 WHERE (`id` = '3');
```

**Результат:** благодаря ограничению **`ON UPDATE CASCADE`** операция пройдёт успешно. СУБД:

1. Изменит значение `id` пользователя с `3` на `100` в таблице `users`.
2. **Автоматически обновит** значение `userid` с `3` на `100` во всех связанных записях таблицы `resume`.

Таким образом, связь между пользователем и его резюме сохранится, несмотря на изменение первичного ключа.

> *Примечание:* на практике первичные ключи стараются не изменять — это нарушает стабильность ссылок в связанных таблицах и может привести к неожиданному каскадному обновлению большого количества записей. Вместо изменения `id` обычно создают новую запись и удаляют старую.
